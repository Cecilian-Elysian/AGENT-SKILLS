# 文档协作撰写工作流程

此技能提供结构化工作流程，引导用户完成协作文档创建。作为积极指南，引导用户完成三个阶段：上下文收集、完善与结构、读者测试。

## 何时提供此工作流程

**触发条件：**
- 用户提及撰写文档："写一份文档"、"起草提案"、"创建规范"、"写下来"
- 用户提及特定文档类型："PRD"、"设计文档"、"决策文档"、"RFC"
- 用户似乎正在开始一项重要的写作任务

**初始提议：**
向用户提供协作撰写文档的结构化工作流程。解释三个阶段：

1. **上下文收集**：用户提供所有相关上下文，Claude 提出澄清问题
2. **完善与结构**：通过头脑风暴和编辑迭代构建每个部分
3. **读者测试**：用新的 Claude（无上下文）测试文档，在他人阅读前发现盲点

解释这种方法有助于确保文档在他人阅读时效果良好（包括当他们将其粘贴到 Claude 中）。询问他们是否想尝试此工作流程或更喜欢自由形式。

如果用户拒绝，自由形式工作。如果用户接受，继续第一阶段。

## 第一阶段：上下文收集

**目标：**缩小用户所知与 Claude 所知之间的差距，使后续智能指导成为可能。

### 初始问题

首先询问用户关于文档的元上下文：

1. 这是什么类型的文档？（如技术规范、决策文档、提案）
2. 主要受众是谁？
3. 当有人阅读时，希望产生什么影响？
4. 有模板或特定格式要遵循吗？
5. 还有其他约束或上下文需要了解吗？

告知他们可以用简写或任何方式_dump 信息来回答。

**如果用户提供模板或提及文档类型：**
- 询问他们是否有模板文档可以分享
- 如果他们提供共享文档链接，使用适当的集成来获取它
- 如果他们提供文件，阅读它

**如果用户提及编辑现有共享文档：**
- 使用适当的集成读取当前状态
- 检查没有 alt 文本的图像
- 如果存在没有 alt 文本的图像，解释当他人使用 Claude 理解文档时，Claude 将无法看到它们。询问是否要生成 alt 文本。如果是，请求他们将每个图像粘贴到聊天中以生成描述性 alt 文本。

### 信息倾倒

一旦初始问题得到回答，鼓励用户倾倒他们拥有的所有上下文。请求信息如：
- 项目/问题的背景
- 相关团队讨论或共享文档
- 为什么没有使用替代解决方案
- 组织上下文（团队动态、过去事件、政治）
- 时间压力或约束
- 技术架构或依赖
- 利益相关者关注

建议他们不要担心组织——只是全部倾倒出来。提供多种提供上下文的方式：
- 意识流信息倾倒
- 指向团队频道或线程以阅读
- 链接到共享文档

**如果有可用的集成**（如 Slack、Teams、Google Drive、SharePoint 或其他 MCP 服务器），提及这些可用于直接拉取上下文。

**如果没有检测到集成且在 Claude.ai 或 Claude 应用中：**建议他们可以在 Claude 设置中启用连接器，以允许直接从消息应用和文档存储拉取上下文。

告知他们一旦完成初始倾倒，将提出澄清问题。

**在上下文收集期间：**

- 如果用户提及团队频道或共享文档：
  - 如果集成可用：告知他们内容将被现在读取，然后使用适当的集成
  - 如果集成不可用：解释缺乏访问。建议他们在 Claude 设置中启用连接器，或直接粘贴相关内容。

- 如果用户提及未知实体/项目：
  - 询问是否应搜索连接的工具有更多信息
  - 在搜索前等待用户确认

- 当用户提供上下文时，跟踪所学到的和仍不清楚的

**提出澄清问题：**

当用户表示已完成初始倾倒（或提供了大量上下文后），提出澄清问题以确保理解：

根据上下文中的空白生成 5-10 个编号问题。

告知他们可以用简写回答（如"1: 是，2: 见 #频道，3: 不因为向后兼容"），链接到更多文档，指向要阅读的频道，或只是继续信息倾倒。无论对他们最有效的方式是什么。

**退出条件：**
当问题显示理解时——当可以询问边缘情况和权衡而无需解释基础知识时——收集了足够的上下文。

**过渡：**
询问是否在此阶段要提供更多上下文，还是时候进入起草文档。

如果用户想添加更多，让他们添加。准备好后，继续第二阶段。

## 第二阶段：完善与结构

**目标：**通过头脑风暴、策划和迭代完善逐节构建文档。

**给用户的说明：**
解释文档将逐节构建。对于每个部分：
1. 将提出关于要包含什么的澄清问题
2. 将头脑风暴 5-20 个选项
3. 用户将指示保留/删除/合并什么
4. 将起草该部分
5. 将通过精细编辑进行完善

从未知最多的部分开始（通常是核心决策/提案），然后完成其余部分。

**部分排序：**

如果文档结构清楚：
询问他们想从哪个部分开始。

建议从未知最多的部分开始。对于决策文档，那是核心提案。对于规范，通常是技术方法。摘要部分最好留到最后。

如果用户不知道需要哪些部分：
根据文档类型和模板，建议适合该文档类型的 3-5 个部分。

询问此结构是否有效，或者是否要调整。

**一旦结构达成一致：**

创建包含所有部分的占位符文本的初始文档结构。

**如果有对工件的访问权限：**
使用 `create_file` 创建工件。这为 Claude 和用户提供了一个可以工作的脚手架。

告知他们将创建包含所有部分标题和简要占位符文本（如"[待撰写]"或"[此处内容]"）的初始结构。

提供脚手架链接并指示是时候填充每个部分了。

**如果没有对工件的访问权限：**
在工作目录中创建 markdown 文件。适当命名（如 `decision-doc.md`、`technical-spec.md`）。

告知他们将创建包含所有部分标题和占位符文本的初始结构。

确认文件名已创建并指示是时候填充每个部分了。

**对于每个部分：**

### 步骤 1：澄清问题

宣布开始 [部分名称] 部分。提出 5-10 个关于应包含什么的澄清问题：

根据上下文和部分目的生成 5-10 个具体问题。

告知他们可以用简写回答或只是指示什么是重要的。

### 步骤 2：头脑风暴

对于 [部分名称] 部分，根据部分的复杂性头脑风暴 [5-20] 个可能包含的内容。寻找：
- 可能被遗忘的共享上下文
- 尚未提及的角度或考虑

根据部分复杂性生成 5-20 个编号选项。最后，如果他们想要更多选项，提供更多头脑风暴。

### 步骤 3：策划

询问哪些要点应该保留、删除或合并。请求简要理由以帮助了解后续部分的优先级。

提供示例：
- "保留 1,4,7,9"
- "删除 3（与 1 重复）"
- "删除 6（受众已经知道这个）"
- "合并 11 和 12"

**如果用户给出自由形式反馈**（如"看起来不错"或"我大部分喜欢但..."）而不是编号选择，提取他们的偏好并继续。解析他们想要保留/删除/更改的内容并应用它。

### 步骤 4：空白检查

根据他们选择的内容，询问 [部分名称] 部分是否有重要遗漏。

### 步骤 5：起草

使用 `str_replace` 将该部分的占位符文本替换为实际起草的内容。

宣布 [部分名称] 部分将根据他们选择的起草。

**如果使用工件：**
起草后，提供工件链接。

请他们阅读并指示要更改什么。注意具体性有助于为后续部分学习。

**如果使用文件（无工件）：**
起草后，确认完成。

告知他们 [部分名称] 部分已在 [文件名] 中起草。请他们阅读并指示要更改什么。注意具体性有助于为后续部分学习。

**给用户的关键说明（起草第一部分时包括）：**
提供说明：不要直接编辑文档，而是请他们指示要更改什么。这有助于了解他们的风格以备后续部分。例如："删除 X 项目——已被 Y 覆盖"或"使第三段更简洁"。

### 步骤 6：迭代完善

当用户提供反馈时：
- 使用 `str_replace` 进行编辑（从不重印整个文档）
- **如果使用工件：**每次编辑后提供工件链接
- **如果使用文件：**仅确认编辑完成
- 如果用户直接编辑文档并要求读取：心理记下他们所做的更改，并在后续部分牢记（这显示他们的偏好）

**继续迭代**直到用户对该部分满意。

### 质量检查

经过 3 次连续迭代且没有实质性更改后，询问是否可以在不丢失重要信息的情况下删除任何内容。

当部分完成时，确认 [部分名称] 完成。询问是否准备好进入下一部分。

**重复所有部分。**

### 接近完成

当接近完成（80%+ 的部分完成），宣布打算重新阅读整个文档并检查：
- 整个部分的一致性和流程
- 冗余或矛盾
- 任何看起来像"敷衍"或通用填充物的东西
- 每个句子是否都有分量

阅读整个文档并提供反馈。

**当所有部分都已起草和完善：**
宣布所有部分都已起草。表示打算再审查一次完整文档。

审查整体连贯性、流程和完整性。

提供任何最终建议。

询问是否准备好进入读者测试，或者是否还有其他要完善的。

## 第三阶段：读者测试

**目标：**用新的 Claude（无上下文渗透）测试文档，验证对读者是否有效。

**给用户的说明：**
解释现在将进行测试，看看文档对读者是否真的有效。这发现了盲点——对作者有意义但可能使他人困惑的东西。

### 测试方法

**如果有对子代理的访问权限**（如在 Claude Code 中）：

直接进行测试，无需用户参与。

### 步骤 1：预测读者问题

宣布打算预测读者在尝试发现此文档时可能提出的问题。

生成读者会现实提出的 5-10 个问题。

### 步骤 2：用子代理测试

宣布这些将由新的 Claude 实例测试（无此对话中的上下文）。

对于每个问题，使用仅文档内容和问题调用子代理。

总结 Reader Claude 每个问题的对/错。

### 步骤 3：运行额外检查

宣布将执行额外检查。

调用子代理检查歧义、错误假设、矛盾。

总结发现的问题。

### 步骤 4：报告和修复

如果发现问题：
报告 Reader Claude 在特定问题上遇到困难。

列出具体问题。

表示打算修复这些空白。

返回有问题部分的重完善。

---

**如果没有对子代理的访问权限**（如 claude.ai Web 界面）：

用户需要手动进行测试。

### 步骤 1：预测读者问题

询问人们在尝试发现此文档时可能问什么问题。他们会输入到 Claude.ai 什么？

生成读者会现实提出的 5-10 个问题。

### 步骤 2：设置测试

提供测试说明：
1. 打开新的 Claude 对话：https://claude.ai
2. 粘贴或分享文档内容（如果使用启用了连接器的共享文档平台，提供链接）
3. 向 Reader Claude 提出生成的问题

对于每个问题，指导 Reader Claude 提供：
- 答案
- 是否有歧义或不清楚的地方
- 文档假设读者已经知道什么知识/上下文

检查 Reader Claude 是否给出正确答案或误解了什么。

### 步骤 3：额外检查

也询问 Reader Claude：
- "文档中什么可能对读者是歧义或不清楚的？"
- "文档假设读者已经有什么知识或上下文？"
- "有内部矛盾或不一致吗？"

### 步骤 4：根据结果迭代

询问 Reader Claude 错了什么或在哪里遇到困难。表示打算修复这些空白。

返回任何有问题部分的重完善。

---

### 退出条件（两种方法）

当 Reader Claude 持续正确回答问题且未发现新空白或歧义时，文档已就绪。

## 最终审查

当读者测试通过时：
宣布文档已通过 Reader Claude 测试。完成前：

1. 建议他们自己进行最终阅读——他们拥有此文档并对其质量负责
2. 建议仔细检查任何事实、链接或技术细节
3. 请他们验证是否达到了他们想要的影响

询问他们是否想要再审查一次，或者工作是否完成。

**如果用户想要最终审查，提供它。否则：**
宣布文档完成。提供一些最终提示：
- 考虑在附录中链接此对话，以便读者可以看到文档是如何开发的
- 使用附录提供深度而不膨胀主文档
- 随着从真实读者收到反馈更新文档

## 有效指导的提示

**语气：**
- 直接和程序化
- 简要解释影响用户行为的理由
- 不要试图"推销"方法——只是执行

**处理偏差：**
- 如果用户想跳过某个阶段：询问他们是否想跳过这个并自由撰写
- 如果用户似乎感到沮丧：承认这比预期花费更长时间。建议加快速度的方法
- 始终给予用户调整过程的能动性

**上下文管理：**
- 在整个过程中，如果提到的内容缺少上下文，主动询问
- 不要让空白积累——随着出现解决

**工件管理：**
- 使用 `create_file` 起草完整部分
- 使用 `str_replace` 进行所有编辑
- 每次更改后提供工件链接
- 永远不要用工件进行头脑风暴列表——那只是对话

**质量优于速度：**
- 不要匆忙完成阶段
- 每次迭代都应有有意义的改进
- 目标是创建对读者真正有效的文档
