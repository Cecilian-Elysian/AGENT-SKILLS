# 技能创建器

用于创建新技能和迭代改进它们的技能。

从高层次来看，创建技能的过程如下：

- 决定技能想要做什么以及大致如何做
- 撰写技能草稿
- 创建一些测试提示并在其上运行 claude-with-access-to-the-skill
- 帮助用户定性和定量评估结果
  - 当运行在后台发生时，如果没有定量评估则起草一些（如果有，可以按原样使用或在你觉得需要更改时修改）。然后向用户解释它们（或者如果已经存在，解释已经存在的）
  - 使用 `eval-viewer/generate_review.py` 脚本向用户展示结果供他们查看，同时也让他们查看定量指标
- 根据用户对结果的评估反馈重写技能（以及如果从定量基准测试中发现明显缺陷）
- 重复直到满意
- 扩展测试集并再次大规模尝试

当使用此技能时，你的工作是弄清楚用户在这个过程的哪个阶段，然后帮助他们完成这些阶段。例如，也许他们会说"我想为 X 制作一个技能"。你可以帮助缩小他们的意思，撰写草稿，撰写测试用例，弄清楚他们想要如何评估，运行所有提示，并重复。

另一方面，也许他们已经有了技能草稿。在这种情况下，你可以直接进入评估/迭代循环部分。

当然，你应该始终保持灵活性，如果用户说"我不需要运行一堆评估，只是和我一起放松"，你可以改为那样做。

然后在技能完成后（但同样，顺序是灵活的），你也可以运行技能描述改进器，我们有一个单独的脚本，用于优化技能的触发。

好的？好的。

## 与用户沟通

技能创建器可能被具有各种编码术语熟悉程度的用户使用。如果你不熟悉（你怎么能，它只是最近才开始），现在有一个趋势，Claude 的力量激发水管工打开他们的终端，父母和祖父母搜索"如何安装 npm"。另一方面，大多数用户可能相当懂电脑。

所以请注意上下文线索以了解如何措辞你的沟通！在默认情况下，给你一些概念：

- "evaluation" 和 "benchmark" 是边界情况，但可以接受
- 对于"JSON" 和"assertion"，你希望在看到用户的认真线索表明他们知道这些东西是什么后再使用它们而不解释

如果有疑问，简要解释术语是可以的，如果你不确定用户是否会理解，请随意澄清术语。

## 创建技能

### 捕获意图

首先理解用户的意图。当前对话可能已经包含用户想要捕获的工作流程（例如，他们说"把这个变成一个技能"）。如果是，首先从对话历史中提取答案——使用的工具、步骤顺序、用户做出的更正、观察到的输入/输出格式。用户可能需要填补空白，并在继续下一步之前确认。

1. 这个技能应该让 Claude 能够做什么？
2. 什么时候应该触发这个技能？（什么用户措辞/上下文）
3. 预期的输出格式是什么？
4. 我们应该设置测试用例来验证技能有效吗？具有客观可验证输出的技能（文件转换、数据提取、代码生成、固定工作流程步骤）受益于测试用例。具有主观输出的技能（写作风格、艺术）通常不需要它们。根据技能类型建议适当的默认值，但让用户决定。

### 访谈和研究

主动询问有关边缘情况、输入/输出格式、示例文件、成功标准和依赖项的问题。在撰写测试提示之前等待这部分完善。

检查可用的 MCP——如果对研究有用（搜索文档、查找类似技能、查找最佳实践），如果有则通过子代理并行研究，否则内联。准备好上下文以减少用户负担。

### 撰写 SKILL.md

根据用户访谈，填写这些组件：

- **name**：技能标识符
- **description**：何时触发，做什么。这是主要的触发机制——包括技能做什么以及何时使用的具体上下文。所有"何时使用"信息放在这里，不在正文中。注意：目前 Claude 有倾向于"触发不足"技能——在它们有用时不使用它们。为了解决这个问题，请将技能描述写得稍微"有推动性"。例如，不要写"How to build a simple fast dashboard to display internal Anthropic data."，你可能写"How to build a simple fast dashboard to display internal Anthropic data. Make sure to use this skill whenever the user mentions dashboards, data visualization, internal metrics, or wants to display any kind of company data, even if they don't explicitly ask for a 'dashboard.'"
- **compatibility**：必需工具、依赖项（可选，很少需要）
- **其余的技能内容 :)**

### 技能撰写指南

#### 技能剖析

```
skill-name/
├── SKILL.md (必需)
│   ├── YAML frontmatter (name, description 必需)
│   └── Markdown 说明
└── 捆绑资源 (可选)
    ├── scripts/    - 用于确定性/重复性任务的可执行代码
    ├── references/ - 根据需要加载到上下文的文档
    └── assets/     - 输出中使用的文件（模板、图标、字体）
```

#### 渐进式披露

技能使用三级加载系统：
1. **元数据**（name + description）- 始终在上下文中（~100 词）
2. **SKILL.md 正文** - 技能触发时在上下文中（理想 <500 行）
3. **捆绑资源** - 根据需要（无限，脚本可以执行而不加载）

这些字数是近似的，你可以随意更长如果需要。

**关键模式：**
- 保持 SKILL.md 在 500 行以下；如果你接近这个限制，添加额外层次结构以及关于模型使用技能应该去哪里遵循的清晰指针
- 从 SKILL.md 清晰引用文件并指导何时阅读它们
- 对于大参考文件（>300 行），包含目录

**领域组织**：当技能支持多个领域/框架时，按变体组织：
```
cloud-deploy/
├── SKILL.md (工作流程 + 选择)
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
Claude 只读取相关参考文件。

#### 无意外原则

不言而喻，技能不能包含恶意软件、利用代码或任何可能损害系统安全的内容。技能的内容如果被描述不应该使用户惊讶。不要配合创建误导性技能或设计促进未授权访问、数据外泄或其他恶意活动的技能。像"扮演 XYZ"之类的事情是可以的。

#### 撰写模式

在说明中使用祈使语气。

**定义输出格式** - 你可以这样做：
```markdown
## 报告结构
始终使用此确切模板：
# [标题]
## 执行摘要
## 主要发现
## 建议
```

**示例模式** - 包括示例很有用。你可以这样格式化（但如果"输入"和"输出"在示例中，你可能想稍微偏离）：
```markdown
## 提交消息格式
**示例 1:**
输入：使用 JWT 令牌添加用户认证
输出：feat(auth): implement JWT-based authentication
```

### 撰写风格

尝试向模型解释为什么事情是重要的，而不是使用沉重的 MUST。用心智理论尝试使技能通用而不是超级狭窄到特定示例。首先撰写草稿，然后用新鲜眼光查看并改进它。

### 测试用例

撰写技能草稿后，提出 2-3 个现实的测试提示——用户实际会说的事物类型。与用户分享：[你不必使用完全相同的语言]"这是我想尝试的几个测试用例。这些看起来对吗，或者你想添加更多？"然后运行它们。

将测试用例保存到 `evals/evals.json`。还不要写断言——只是提示。你会在下一步（当运行正在进行时）起草断言。

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "用户的任务提示",
      "expected_output": "预期结果描述",
      "files": []
    }
  ]
}
```

参见 `references/schemas.md` 获取完整 schema（包括稍后添加的 `assertions` 字段）。

## 运行和评估测试用例

本节是一个连续序列——不要中途停止。不要使用 `/skill-test` 或任何其他测试技能。

将结果放在作为技能目录兄弟的 `<skill-name>-workspace/` 中。在工作区内，按迭代（`iteration-1/`、`iteration-2/` 等）组织，在其中每个测试用例有一个目录（`eval-0/`、`eval-1/` 等）。不要预先创建所有这些——只是随进度创建目录。

### 步骤 1：在同一轮中生成所有运行（有技能 AND 基线）

对于每个测试用例，在同一轮中生成两个子代理——一个有技能，一个没有。这很重要：不要先生成有技能运行然后稍后回来生成基线。立即启动所有内容以便它们都差不多同时完成。

**有技能运行：**

```
执行此任务：
- 技能路径：<path-to-skill>
- 任务：<eval prompt>
- 输入文件：<eval 文件如果有，或"无">
- 保存输出到：<workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 要保存的输出：<用户关心的内容——例如".docx 文件"、"最终 CSV">
```

**基线运行**（相同提示，但基线取决于上下文）：
- **创建新技能**：根本没有技能。相同提示，无技能路径，保存到 `without_skill/outputs/`。
- **改进现有技能**：旧版本。编辑前快照技能（`cp -r <skill-path> <workspace>/skill-snapshot/`），然后将基线子代理指向快照。保存到 `old_skill/outputs/`。

为每个测试用例编写 `eval_metadata.json`（断言可以暂时为空）。给每个评估一个描述性名称基于它测试什么——不仅仅是"eval-0"。也用这个名称作为目录名。如果此迭代使用新的或修改的评估提示，为每个新评估目录创建这些文件——不要假设它们从前一个迭代继承。

```json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "用户的任务提示",
  "assertions": []
}
```

### 步骤 2：在运行进行时，起草断言

不要只是等待运行完成——你可以高效利用这段时间。为每个测试用例起草定量断言并向用户解释。如果 `evals/evals.json` 中已存在断言，审查它们并解释它们检查什么。

好的断言是客观可验证的并且有描述性名称——它们应该在基准查看器中清晰阅读，以便某人浏览结果立即理解每个检查什么。具有主观技能的技能（写作风格、设计质量）更适合定性评估——不要将断言强加于需要人类判断的事情上。

一旦起草，更新 `eval_metadata.json` 文件和 `evals/evals.json` 以及断言。还要向用户解释他们将在查看器中看到什么——定性输出和定量基准。

### 步骤 3：当运行完成时，捕获计时数据

当每个子代理任务完成时，你收到包含 `total_tokens` 和 `duration_ms` 的通知。立即将此数据保存到运行目录中的 `timing.json`：

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

这是捕获此数据的唯一机会——它通过任务通知传来，不会被其他地方保留。当收到每个通知时处理它，而不是试图批量处理。

### 步骤 4：评分、聚合并启动查看器

所有运行完成后：

1. **为每个运行评分** — 生成评分子代理（或内联评分）读取 `agents/grader.md` 并针对输出评估每个断言。将结果保存到每个运行目录中的 `grading.json`。grading.json 期望数组必须使用字段 `text`、`passed` 和 `evidence`（不是 `name`/`met`/`details` 或其他变体）——查看器依赖这些确切的字段名。对于可以编程检查的断言，编写并运行脚本而不是用眼检查——脚本更快、更可靠，可以跨迭代重用。

2. **聚合到基准** — 从 skill-creator 目录运行聚合脚本：
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```
   这会产生 `benchmark.json` 和 `benchmark.md`，包含每个配置及其均值 ± 标准差和 delta 的通过率、时间和令牌。如果手动生成 benchmark.json，参见 `references/schemas.md` 获取查看器期望的确切 schema。
将每个有技能版本放在其基线对应之前。

3. **做分析师通过** — 阅读基准数据并揭示聚合统计可能隐藏的模式。参见 `agents/analyzer.md`（"分析基准结果"部分）了解要寻找什么——像总是通过的断言（无区分）、高方差评估（可能不稳定）和时间/令牌权衡。

4. **启动带有定性输出和定量数据的查看器**：
   ```bash
   nohup python <skill-creator-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   对于迭代 2+，也传递 `--previous-workspace <workspace>/iteration-<N-1>`。

   **同事/无头环境：** 如果 `webbrowser.open()` 不可用或环境没有显示器，使用 `--static <output_path>` 写入独立 HTML 文件而不是启动服务器。当用户点击"提交所有评论"时，反馈将下载为 `feedback.json` 文件。下载后，将 `feedback.json` 复制到工作区目录以便下次迭代拾取。

注意：请使用 generate_review.py 创建查看器；不需要编写自定义 HTML。

5. **告诉用户** 类似这样的话："我已在浏览器中打开结果。有两个选项卡——'输出'让你点击每个测试用例并留下反馈，'基准'显示定量比较。你完成后，回来这里告诉我。"

### 用户在查看器中看到的内容

"输出"选项卡一次显示一个测试用例：
- **提示**：给出的任务
- **输出**：技能产生的文件，尽可能内联渲染
- **上一个输出**（迭代 2+）：显示上次迭代输出的折叠部分
- **正式评分**（如果运行了评分）：显示断言通过/失败的折叠部分
- **反馈**：输入时自动保存的文本框
- **上一个反馈**（迭代 2+）：他们在上次输入的评论，显示在文本框下方

"基准"选项卡显示统计摘要：每个配置的通过率、时间和令牌使用，以及每个评估细分和分析师观察。

导航通过上一个/下一个按钮或箭头键。完成后，他们点击"提交所有评论"，将所有反馈保存到 `feedback.json`。

### 步骤 5：阅读反馈

当用户告诉你完成时，读取 `feedback.json`：

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "图表缺少轴标签", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "完美，我喜欢这个", "timestamp": "..."}
  ],
  "status": "complete"
}
```

空反馈意味着用户认为它没问题。将改进集中在用户有具体抱怨的测试用例上。

完成后终止查看器服务器：

```bash
kill $VIEWER_PID 2>/dev/null
```

## 改进技能

这是循环的核心。你已经运行了测试用例，用户已经审查了结果，现在你需要根据他们的反馈使技能变得更好。

### 如何思考改进

1. **从反馈中泛化。** 这里发生的大画面事情是，我们正在尝试创建可以跨许多不同提示使用一百万次（也许 literally，也许更多谁知道）的技能。你和用户只是迭代少数例子因为它有助于更快。用户非常了解这些例子，他们可以快速评估新输出。但是如果你和用户共同开发的技能只适用于那些例子，它就没用了。不要进行挑剔的过拟合更改，或者压迫性限制性的 MUST，如果有顽固问题，尝试分支出来使用不同的隐喻，或者推荐不同的工作模式。尝试相对便宜，也许你会遇到很棒的东西。

2. **保持提示精简。** 删除不起作用的内容。确保阅读记录而不只是最终输出——如果看起来技能让模型浪费大量时间做无成效的事情，你可以尝试删除让它那样做的技能部分并看看会发生什么。

3. **解释原因。** 努力解释你要求模型做的每件事的**原因**。今天的 LLM 很聪明。他们有良好的心智理论，当给予良好框架时可以超越死板指令真正发挥作用。即使用户反馈简洁或沮丧，试着真正理解任务和用户为什么写他们写的，以及他们实际写了什么，然后将这种理解传递到说明中。如果你发现自己用全大写写 ALWAYS 或 NEVER，或者使用超级僵化结构，那是黄旗——如果可能，重新框架并解释原因，以便模型理解为什么你要求的事情是重要的。这是一种更人性化、更强大的方法。

4. **寻找跨测试用例的重复工作。** 阅读测试运行的记录，注意子代理是否都独立编写了类似的辅助脚本或对某事物采用相同的多步骤方法。如果所有 3 个测试用例都导致子代理编写 `create_docx.py` 或 `build_chart.py`，那是强烈信号技能应该捆绑那个脚本。写一次，放在 `scripts/`，并告诉技能使用它。这节省了每次未来调用重新发明轮子。

这个任务非常重要（我们正试图每年在这里创造数十亿美元的经济价值！）你的思考时间不是瓶颈；花时间真正思考。我建议撰写修订草稿然后用新的眼光看它并改进。真正尽力进入用户的头脑，理解他们想要和需要什么。

### 迭代循环

改进技能后：

1. 将你的改进应用到技能
2. 将所有测试用例重新运行到新的 `iteration-<N+1>/` 目录，包括基线运行。如果创建新技能，基线始终是 `without_skill`（无技能）——跨迭代保持不变。如果改进现有技能，使用你的判断关于什么是合理的基线：用户带来的原始版本，或者上一个迭代。
3. 启动审查器，传递 `--previous-workspace` 指向上一迭代
4. 等待用户审查并告诉你完成
5. 读取新反馈，再改进，重复

继续直到：
- 用户说他们满意
- 反馈都是空的（一切看起来都好）
- 你没有取得有意义的进展

## 高级：盲测比较

对于想要更严格比较技能两个版本的情况（如用户问"新版本实际上更好吗？"），有盲测系统。阅读 `agents/comparator.md` 和 `agents/analyzer.md` 了解详情。基本思想是：将两个输出交给独立代理而不告诉它哪个是哪个，让它判断质量。然后分析为什么赢家赢了。

这是可选的，需要子代理，大多数用户不需要。人类审查循环通常就足够了。

## 描述优化

SKILL.md frontmatter 中的 description 字段是决定 Claude 是否调用技能的主要机制。创建或改进技能后，提供优化描述以提高触发准确性。

### 步骤 1：生成触发评估查询

创建 20 个评估查询——应该是触发和不应该触发的混合。保存为 JSON：

```json
[
  {"query": "用户提示", "should_trigger": true},
  {"query": "另一个提示", "should_trigger": false}
]
```

查询必须是现实的，是 Claude Code 或 Claude.ai 用户实际会输入的东西。不是抽象请求，而是具体且详细的请求。例如，文件路径、用户工作或情况的个人上下文、列名和值、公司名称、URL。一点背景故事。有些可能是小写或包含缩写或拼写错误或随意话语。使用不同长度的混合，关注边缘情况而不是使它们显而易见（用户将有机会签署）。

差： `"Format this data"`, `"Extract text from PDF"`, `"Create a chart"`

好： `"ok so my boss just sent me this xlsx file (its in my downloads, called something like 'Q4 sales final FINAL v2.xlsx') and she wants me to add a column that shows the profit margin as a percentage. The revenue is in column C and costs are in column D i think"`

对于**应该触发**的查询（8-10），考虑覆盖率。你想要同一意图的不同措辞——有些正式，有些随意。包括用户没有明确命名技能或文件类型但显然需要它的情况。加入一些不常见的用例以及这个技能与其他技能竞争但应该赢的情况。

对于**不应该触发**的查询（8-10），最有价值的是near-misses——共享关键词或技能概念但实际需要不同东西的查询。考虑相邻领域、歧义措辞（朴素关键词匹配会触发但不应该），以及查询涉及技能做什么但在另一个工具更合适的上下文中。

要避免的关键：不要使不应该触发的查询明显无关。"Write a fibonacci function" 作为 PDF 技能的负面测试太容易——它不测试任何东西。负面情况应该是真正棘手的。

### 步骤 2：与用户审查

使用 HTML 模板向用户展示评估集以供审查：

1. 从 `assets/eval_review.html` 读取模板
2. 替换占位符：
   - `__EVAL_DATA_PLACEHOLDER__` → 评估项的 JSON 数组（周围不带引号——它是 JS 变量赋值）
   - `__SKILL_NAME_PLACEHOLDER__` → 技能名称
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → 技能的当前描述
3. 写入临时文件（如 `/tmp/eval_review_<skill-name>.html`）并打开：`open /tmp/eval_review_<skill-name>.html`
4. 用户可以编辑查询、切换 should-trigger、添加/删除条目，然后点击"导出评估集"
5. 文件下载到 `~/Downloads/eval_set.json`——检查下载文件夹中最新版本以防有多个（如 `eval_set (1).json`）

这一步很重要——糟糕的评估查询导致糟糕的描述。

### 步骤 3：运行优化循环

告诉用户："这需要一些时间——我会在后台运行优化循环并定期检查它。"

将评估集保存到工作区，然后在后台运行：

```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <model-id-powering-this-session> \
  --max-iterations 5 \
  --verbose
```

使用系统提示中的模型 ID（为本次会话提供动力的）以便触发测试匹配用户实际体验。

当它运行时，定期尾随输出以向用户提供关于它在哪个迭代以及分数看起来怎样的更新。

这自动处理完整优化循环。它将评估集分成 60% 训练和 40% 保留测试，评估当前描述（运行每个查询 3 次以获得可靠触发率），然后调用 Claude 根据失败的提出改进。它在训练和测试上重新评估每个新描述，迭代最多 5 次。当完成时，在浏览器中打开显示每个迭代结果的 HTML 报告并返回带有 `best_description` 的 JSON——由测试分数而不是训练分数选择以避免过拟合。

### 技能触发的工作原理

理解触发机制有助于设计更好的评估查询。技能出现在 Claude 的 `available_skills` 列表中及其名称 + 描述，Claude 根据该描述决定是否咨询技能。需要知道的重要事情是，Claude 只为它不能轻松处理的任务咨询技能——简单、一步查询如"read this PDF"可能不会触发技能即使描述完全匹配，因为 Claude 可以直接用基本工具处理它们。复杂、多步或专业查询在描述匹配时可靠触发技能。

这意味着你的评估查询应该足够实质，以便 Claude 实际受益于咨询技能。像"read file X"这样的简单查询是糟糕的测试用例——它们不会触发技能无论描述质量如何。

### 步骤 4：应用结果

从 JSON 输出中获取 `best_description` 并更新技能的 SKILL.md frontmatter。先向用户展示之前/之后并报告分数。

### 打包和呈现（仅当 `present_files` 工具可用时）

检查你是否有权访问 `present_files` 工具。如果没有，跳过此步骤。如果有，打包技能并向用户呈现 .skill 文件：

```bash
python -m scripts.package_skill <path/to/skill-folder>
```

打包后，将用户引导到生成的 .skill 文件路径，以便他们可以安装它。

## Claude.ai 特定说明

在 Claude.ai 中，核心工作流程相同（草稿 → 测试 → 审查 → 改进 → 重复），但因为 Claude.ai 没有子代理，一些机制改变。以下是要适应的内容：

**运行测试用例**：没有子代理意味着没有并行执行。对于每个测试用例，阅读技能的 SKILL.md，然后按照其说明自己完成测试提示。一次一个。这不如独立子代理严格（你写了技能也在运行它，所以你完全了解上下文），但这是有用的健全性检查——人类审查步骤弥补了。没有基线运行——只是使用技能按请求完成任务。

**审查结果**：如果你不能打开浏览器（如 Claude.ai 的 VM 没有显示器，或者你在远程服务器上），完全跳过浏览器审查器。相反，在对话中直接呈现结果。对于每个测试用例，显示提示和输出。如果输出是用户需要查看的文件（如 .docx 或 .xlsx），保存到文件系统并告诉他们在哪里以便下载和检查。内联请求反馈："这看起来怎么样？有什么要改的吗？"

**基准测试**：跳过定量基准测试——它依赖基线比较在没有子代理时没有意义。专注于用户的定性反馈。

**迭代循环**：相同——改进技能，重新运行测试用例，请求反馈——只是没有中间的浏览器审查器。如果有文件系统，你仍然可以按迭代目录组织结果。

**描述优化**（`run_loop.py` / `run_eval.py`）：此部分需要 `claude` CLI 工具（特别是 `claude -p`），仅在 Claude Code 中可用。在 Claude.ai 上跳过它。

**盲测比较**：需要子代理。跳过它。

**打包**：`package_skill.py` 脚本在任何有 Python 和文件系统的地方都有效。在 Claude.ai 上，你可以运行它，用户可以下载生成的 .skill 文件。

**更新现有技能**：用户可能要求你更新现有技能而不是创建新技能。在这种情况下：
- **保留原始名称。** 注意技能的目录名称和 `name` frontmatter 字段——使用它们不变。例如，如果安装的技能是 `research-helper`，输出 `research-helper.skill`（不是 `research-helper-v2`）。
- **在编辑前复制到可写位置。** 安装的技能路径可能是只读的。复制到 `/tmp/skill-name/`、在那里编辑，然后从副本打包。
- **如果手动打包，先在 `/tmp/` 暂存**，然后复制到输出目录——直接写入可能因权限失败。

## 同事特定说明

如果你在 Cowork，需要知道的主要事情是：

- 你有子代理，所以主要工作流程（并行生成测试用例、运行基线、评分等）都有效。（但是，如果你遇到严重的超时问题，按顺序运行测试提示而不是并行是可以的。）
- 你没有浏览器或显示器，所以在生成评估查看器时，使用 `--static <output_path>` 写入独立 HTML 文件而不是启动服务器。然后提供一个链接，用户可以点击在浏览器中打开 HTML。
- 无论出于什么原因，Cowork 设置似乎让 Claude 不愿冒在运行测试后生成评估查看器，所以只是重申：无论你在 Cowork 还是 Claude Code，在测试后，你应该始终生成评估查看器供人类在你自己评估输入之前查看，使用 `generate_review.py`（不要编写自己的 boutique html 代码）。提前抱歉，但我会全大写：生成评估查看器*在*评估输入*之前*。你想尽快让它们在人类面前！
- 反馈工作不同：因为没有运行服务器，查看器的"提交所有评论"按钮会将 `feedback.json` 下载为文件。然后你可以从那里读取它（你可能需要先请求访问）。
- 打包有效——`package_skill.py` 只需要 Python 和文件系统。
- 描述优化（`run_loop.py` / `run_eval.py`）在 Cowork 中应该完全可以工作，因为它使用 `claude -p` 通过子进程，不是浏览器，但请保存到你完全完成制作技能并用户同意它处于良好状态。
- **更新现有技能**：用户可能要求你更新现有技能而不是创建新技能。按照上面 Claude.ai 部分中的更新指导。
